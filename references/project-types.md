# Шаг 2 — Чеклисты под тип проекта

После 5 универсальных проверок пройди раздел под конкретный тип. Здесь — специфика,
которой нет в общих проверках.

---

## 🌐 Веб-сайт / SPA (Astro, Next.js, Vite, React, Vue, обычный HTML)

- [ ] **Серверные ключи не в клиентском бандле.** Собери прод-бандл и поищи в нём
      секреты: `npm run build && node scripts/secrets-scan.mjs dist` (или `build`/`.next`).
- [ ] **Только `PUBLIC_`/`NEXT_PUBLIC_`/`VITE_` переменные доходят до браузера.**
      Всё остальное — серверное.
- [ ] **CSP и заголовки безопасности** настроены (см. `config-deploy.md`).
- [ ] **Формы** валидируются на сервере; есть защита от спама.
- [ ] **Сторонние скрипты** (аналитика, виджеты) — из доверенных источников, по
      возможности с SRI (`integrity`).
- [ ] **`target="_blank"`** ссылки имеют `rel="noopener"`.
- [ ] **Нет утечки исходников/карт** в проде, если не нужно (`.map` файлы).

---

## 🔌 Бэкенд / API (Express, Fastify, NestJS, Flask, Django, FastAPI)

- [ ] **Аутентификация и владелец ресурса** на каждом эндпоинте (см. `auth-access.md`).
- [ ] **Валидация тела/параметров** строгой схемой.
- [ ] **Rate-limit** на чувствительные эндпоинты.
- [ ] **CORS** — белый список (см. `config-deploy.md`).
- [ ] **Инъекции** (см. `code-vulns.md`) — особое внимание к запросам к БД.
- [ ] **Размер запроса ограничен** (защита от больших тел / DoS).
- [ ] **Ошибки наружу — без стектрейсов.**
- [ ] **Вебхуки проверяют подпись** (Stripe-signature, Telegram secret_token и т.п.).

---

## 🤖 Telegram-бот (telegraf, grammy, aiogram, node-telegram-bot-api)

- [ ] **`BOT_TOKEN` только в `.env`**, не в коде/истории. Утёк токен = чужой полный
      контроль над ботом → 🔴 (ротация через @BotFather → `/revoke`).
- [ ] **Админ-команды проверяют `user_id`/`chat_id`** против белого списка.
      Частая дыра: «админская» команда `/broadcast` доступна любому, кто её знает.
- [ ] **Webhook защищён `secret_token`** (заголовок `X-Telegram-Bot-Api-Secret-Token`),
      чтобы никто не слал поддельные апдейты на твой эндпоинт.
- [ ] **Ввод от пользователя** (текст сообщений) не уходит в `eval`/SQL/команды без
      очистки (см. `code-vulns.md`) — особенно если бот что-то исполняет/запрашивает.
- [ ] **Платежи/действия с деньгами** проверяют сумму и подтверждение на сервере.
- [ ] **Нет логирования токена и личных данных** в открытые логи.
- [ ] **Polling vs webhook:** если webhook — он за HTTPS; если polling — токен всё
      равно в `.env`.

---

## 📱 Мобильное приложение (React Native, Flutter, нативное)

- [ ] **Нет секретов в бинаре.** Всё, что зашито в приложение, извлекаемо. API-ключи
      с правами — только на бэкенде; приложение ходит через свой сервер.
- [ ] **Чувствительные данные — в безопасном хранилище** (Keychain / Keystore),
      не в `AsyncStorage`/`SharedPreferences`/`localStorage` открыто.
- [ ] **Трафик по HTTPS**, certificate pinning для критичных приложений.
- [ ] **Нет отладочных бэкдоров/тестовых эндпоинтов** в релизной сборке.
- [ ] **Разрешения** (камера, геолокация, контакты) — только нужные.
- [ ] **Deeplink/intent** не дают обойти авторизацию.

---

## 🐳 Инфра / DevOps (Docker, CI/CD, Terraform)

- [ ] Полностью пройди CI/CD и Docker-разделы в `config-deploy.md`.
- [ ] **Terraform/IaC:** нет секретов в `.tf` и в `terraform.tfstate` (стейт может
      содержать пароли — хранить в защищённом backend, не в git).
- [ ] **Образы** сканируются (`trivy image ...`), базовые образы свежие.
- [ ] **Доступ к облаку** по ролям с минимальными правами, без долгоживущих
      root-ключей.

---

## 📜 Скрипты / автоматизация (одиночные .py / .js / .sh)

- [ ] **Секреты в `.env`/переменных окружения**, не в теле скрипта.
- [ ] **Command injection:** ввод не склеивается в строку шелла.
- [ ] **Пути к файлам** не из непроверенного ввода (path traversal).
- [ ] **Если скрипт качает/исполняет код из сети** (`curl | bash`, `eval`) — это
      риск; проверяй источник и подпись.
- [ ] **Права файла со скриптом и его доступ к секретам** ограничены.
