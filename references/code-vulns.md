# Проверка 5 — Уязвимости кода (OWASP)

> Дыры в самом коде: когда данные от пользователя попадают туда, где их выполняют
> как команду, запрос или разметку. Ищи маршрут «ввод пользователя → опасное место»
> без очистки по пути.

Метод: найди все точки, где входят пользовательские данные (тело запроса, query,
параметры URL, заголовки, текст сообщения боту, имя файла), и проследи, не уходят
ли они без валидации/экранирования в опасные операции ниже.

## Инъекции

- [ ] **SQL-инъекция.** Запросы собираются конкатенацией строк с пользовательским
      вводом? 🔴. Должно быть: параметризованные запросы / prepared statements /
      ORM с биндингом. Признак дыры: `` `SELECT ... WHERE name='${input}'` ``.
- [ ] **NoSQL-инъекция** (Mongo): пользовательский объект уходит прямо в запрос
      (`{ user: req.body.user }`), позволяя операторы `$gt`/`$ne`. Валидируй типы.
- [ ] **Command injection.** Пользовательский ввод в `exec`/`system`/`os.system`/
      `child_process.exec`? 🔴. Используй `execFile`/`spawn` с массивом аргументов,
      белые списки, без шелла.
- [ ] **Template / SSTI injection.** Ввод подставляется в шаблонизатор как код.

## Клиентские (веб)

- [ ] **XSS.** Пользовательские данные вставляются в HTML без экранирования.
      Признаки: `innerHTML = userData`, `dangerouslySetInnerHTML`, `v-html`,
      `set:html` (Astro) с непроверенным вводом. Экранируй или санитизируй
      (DOMPurify). 🔴 если хранимый XSS (данные сохраняются и показываются другим).
- [ ] **CSP** как второй рубеж против XSS (см. `config-deploy.md`).
- [ ] **Open redirect.** Редирект на URL из параметра без проверки домена.

## Серверные

- [ ] **SSRF.** Сервер ходит по URL, который дал пользователь (загрузка по ссылке,
      вебхуки) — может достучаться до внутренней сети/метаданных облака. Белый список
      доменов, запрет приватных IP.
- [ ] **Path traversal.** Имя файла из ввода идёт в путь (`../../etc/passwd`).
      Нормализуй путь, держи в пределах разрешённой папки.
- [ ] **Небезопасная десериализация.** `pickle.loads`, `yaml.load` (не safe_load),
      `eval`/`Function(...)` над пользовательским вводом → 🔴.
- [ ] **Загрузка файлов.** Проверяй тип/размер/расширение, храни вне webroot,
      генерируй имя сам, не доверяй `Content-Type` от клиента.
- [ ] **Массовое присваивание (mass assignment).** `User.update(req.body)` целиком —
      пользователь может выставить себе `isAdmin=true`. Бери только разрешённые поля.

## Логика и данные

- [ ] **Валидация ввода на сервере** (не только в браузере — клиентскую проверку
      легко обойти). Схемы (zod / pydantic / Joi) на входе.
- [ ] **Числа/суммы/количества** проверяются на отрицательные/переполнение
      (например, цена, баланс, количество товара).
- [ ] **Чувствительные данные не пишутся в логи** (пароли, токены, карты).
- [ ] **Рандом для токенов — криптографический** (`crypto.randomBytes`,
      `secrets`), не `Math.random()`.

## Severity-ориентир

- SQL/command/SSTI-инъекция, небезопасный `eval`/десериализация, хранимый XSS → 🔴.
- Отражённый XSS, SSRF, path traversal, mass assignment → 🔴/🟡 по контексту.
- Нет серверной валидации ввода → 🟡.
- Слабый рандом для неважного → 🟢.

## Как починить (общий принцип)

1. **Никогда не доверяй вводу.** Валидируй на сервере по строгой схеме.
2. **Разделяй данные и код:** параметризованные запросы, `execFile` вместо `exec`,
   экранирование при выводе в HTML.
3. **Белые списки** вместо чёрных (разрешено только перечисленное).
4. Прогоняй статический анализатор под язык (`semgrep`, `bandit` для Python,
   `eslint-plugin-security`) — он ловит шаблоны автоматически.
