# Проверка 1 — Секреты (ключи, токены, пароли)

> Самая частая и самая дорогая ошибка. Утёкший боевой ключ = чужой доступ к твоим
> деньгам, данным, рассылкам. У новичков встречается почти всегда.

## Автосканер

```bash
node scripts/secrets-scan.mjs .
```
Ищет по сигнатурам: AWS, GitHub, OpenAI, Anthropic, Telegram, Slack, Google,
Stripe, Supabase service_role, приватные ключи, пароли, БД-строки с логином/паролем.
Сканирует **и рабочие файлы, и историю git** (ключ мог быть удалён из кода, но
остаться в истории — это всё равно утечка).

## Ручной чеклист (пройди после сканера)

- [ ] **`.env` не закоммичен.** Проверь: `git ls-files | grep -E '^\.env$|\.env\.'`
      Должны быть видны только `.env.example` / `.env.sample` (без реальных значений).
      Если `.env` в репозитории → 🔴.
- [ ] **`.gitignore` содержит** `.env`, `.env.*` (кроме `*.example`), `*.pem`, `*.key`,
      `credentials*`, `*.p12`, `id_rsa`.
- [ ] **Ключи не в клиентском коде.** Во фронтенде (React/Astro/Vue) всё, что видит
      браузер, видит и пользователь. Серверные ключи (`SUPABASE_SERVICE_ROLE_KEY`,
      `OPENAI_API_KEY`, `STRIPE_SECRET`) НЕ должны попадать в клиентский бандл.
      Признак утечки: переменная без префикса `PUBLIC_`/`NEXT_PUBLIC_`/`VITE_`
      используется в компоненте, который рендерится в браузере.
- [ ] **Нет ключей в комментариях, README, тестах, скриншотах, JSON-фикстурах.**
- [ ] **Нет ключей в истории git.** Сканер проверяет; если нашёл — см. ротацию ниже.
- [ ] **Нет приватных ключей** (`-----BEGIN ... PRIVATE KEY-----`) в репозитории.
- [ ] **Боевые и тестовые ключи разделены** (`sk_live_` vs `sk_test_`).

## Как объяснить пользователю (severity)

| Что нашли | Severity | Почему |
|---|---|---|
| Боевой ключ/токен в коде или истории | 🔴 high | Любой, у кого есть доступ к репо (или если репо станет публичным), получает твой доступ |
| `.env` закоммичен с реальными значениями | 🔴 high | То же — секреты уехали в историю |
| Серверный ключ в клиентском бандле | 🔴 high | Виден каждому посетителю сайта |
| Тестовый ключ / слабый хардкод пароля в dev | 🟡 medium | Опасно при переезде в прод |
| Возможный секрет (низкая уверенность) | 🟢 low | Перепроверить вручную |

## Как починить (дай эти шаги пользователю)

**Если ключ утёк — недостаточно просто удалить его из кода. Его уже видели.**

1. **Ротация (заменить ключ).** Зайди в сервис (OpenAI, Telegram @BotFather, Stripe,
   Supabase…) → отзови старый ключ → создай новый. Это обязательный первый шаг.
2. **Убери ключ из кода** → перенеси в `.env`, в коде читай `process.env.X` /
   `import.meta.env.X`. Добавь `.env` в `.gitignore`.
3. **Почисти историю** (если ключ был в коммитах): инструмент `git filter-repo` или
   BFG Repo-Cleaner. Если репо приватный и ключ уже ротирован — можно ограничиться
   ротацией, но публичный репо чистить обязательно.
4. На будущее: подключи pre-commit хук (`gitleaks`) или повторяй этот навык перед
   каждым релизом.
